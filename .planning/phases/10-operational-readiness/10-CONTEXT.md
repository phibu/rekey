<decisions>
## Implementation Decisions

### STAB-018 — `/api/health` dependency coverage (gh#31)

- **D-01 (G1):** Extend the existing `/api/health` endpoint with a nested `checks` object — do NOT introduce a second endpoint. Final shape:
  ```json
  {
    "status": "healthy|degraded|unhealthy",
    "timestamp": "2026-04-17T...",
    "checks": {
      "ad":           { "status": "healthy|degraded|unhealthy", "latency_ms": 12, "last_checked": "..." },
      "smtp":         { "status": "...", "latency_ms": ..., "last_checked": "...", "skipped": false },
      "expiryService":{ "status": "...", "latency_ms": ..., "last_checked": "..." }
    }
  }
  ```
  HTTP 200 when aggregate `status` is `healthy`; HTTP 503 when `degraded` or `unhealthy`. The aggregate rolls up: any `unhealthy` → `unhealthy`; else any `degraded` → `degraded`; else `healthy`.
- **D-02 (G2):** SMTP probe is a TCP connection-open attempt with a 3 s timeout. When `EmailNotificationSettings.Enabled=false` AND `PasswordExpiryNotificationSettings.Enabled=false`, the SMTP check is marked `"skipped": true` and contributes neither to `degraded` nor `unhealthy`. No HELO/TLS handshake (too slow for a health probe; operators get STARTTLS coverage at send time).
- **D-03:** ExpiryService probe reports `healthy` when the background service is either `running` (most recent tick within configured interval + slack) or `not-enabled` (Enabled=false). Otherwise `degraded`. No reflection into `BackgroundService` internals — expose a small `IExpiryServiceDiagnostics` interface similar to `ILockoutDiagnostics`.
- **D-04:** Secrets never appear in the response. Error details (AD bind failure, SMTP connect refused) are recorded in the server log via the existing SIEM pipeline; the health body surfaces a stable status enum only. Already-redacted fields stay redacted.
- **D-05:** All three checks run on the request thread (synchronous) with per-check timeouts. No background caching in this phase — the endpoint is assumed low-traffic and called by load balancers only.

### STAB-019 — Installer post-deploy verification (gh#34)

- **D-06:** After `robocopy`/appsettings-sync + AppPool recycle, `Install-PassReset.ps1` hits `/api/health` and `GET /api/password` on the configured binding.
- **D-07 (G3):** Retry policy: 10 attempts × 2 s interval (≈20 s max wait — matches AppPool cold start). Hard-fail the install with exit code ≥1 if either endpoint is still unreachable or `/api/health` returns 503 after all retries; print the final response body/status line so operators can diagnose.
- **D-08:** Success path: print a green "✓ Health OK — AD: healthy, SMTP: skipped, ExpiryService: not-enabled" summary. No confetti, no emoji in log output (CI-friendly).
- **D-09:** The post-deploy check uses the installer's already-resolved scheme+host+port URL (same source of truth as the "reachable URL" line already printed at install end — STAB-001 output). No new config knobs.
- **D-10:** Respect `-SkipHealthCheck` for operators on air-gapped hosts where outbound loopback is restricted. Default is `$false` — the check runs unless explicitly disabled.

### STAB-020 — CI security scans (gh#35)

- **D-11 (G4):** CI fails on `high` + `critical` npm-audit findings and `High`+`Critical` `dotnet list package --vulnerable` findings. `moderate`/`medium` and below are printed as warnings and do not fail the build. No all-inclusive severity threshold (too noisy given common transitive advisories).
- **D-12:** Exception process — a checked-in `deploy/security-allowlist.json` file lists advisory IDs with rationale + expiration date (forces re-review). CI reads this file and skips those advisories. Unknown advisories always fail; adding to the allowlist requires a PR and expires in ≤90 days.
- **D-13:** Scans run on every push and PR to master, inside the existing `tests.yml` reusable workflow. No separate `security.yml` — keeps CI shape flat and avoids workflow-sprawl.
- **D-14:** Both scans run in a single CI job (`security-audit`) with `continue-on-error: false`. Output is posted as a GitHub Action summary so triage doesn't require log diving.
- **D-15:** GitHub Code Scanning (CodeQL) and Dependabot (already configured per CLAUDE.md) remain independent of this scan — STAB-020 is about PR/push gating, the GitHub-native tools continue to handle periodic deep scans.

### STAB-021 — Password policy visibility (gh#38)

- **D-16 (G5):** `AdPasswordPolicyPanel` already exists (FEAT-002). Scope of STAB-021 is pure UX polish: make the panel visible by default (not behind a disclosure widget if one exists) and ensure it appears above the password fields.
- **D-17:** Do NOT add FGPP (Fine-Grained Password Policy / PSO) resolution. That requires an authenticated AD bind for the specific user BEFORE they submit the form, which is a security posture change — explicitly deferred to v2.0. Known-Limitations.md already documents this trade-off.
- **D-18:** Panel contents continue to pull from `/api/password/policy` (existing endpoint via `usePolicy.ts`). No server changes for STAB-021.
- **D-19:** Accessibility check — panel must be keyboard-navigable and screen-reader-friendly. Use MUI `<Collapse defaultExpanded>` or equivalent so the content is visible from mount without requiring interaction.

<constraints>
## Constraints inherited from upstream phases

- **From Phase 7 (installer):** `-Force` mode must remain silent-non-interactive. Post-deploy verification inherits this — under `-Force`, the health check just runs (no prompt on failure).
- **From Phase 8 (config):** `ConfigSync` + validators already emit 502 on startup fail. `/api/health` is a READINESS probe after successful startup; it does NOT duplicate startup-time validation.
- **From Phase 9 (SIEM):** Health-check failures do NOT emit SIEM events. Health is infra-level; SIEM is user-action-level. Don't add `SiemEventType.HealthCheckFailed`.
- **Platform:** Windows-only (`net10.0-windows`). Health checks must work when domain-joined and when using explicit LDAP creds.
- **Threading:** All new diagnostics interfaces are singleton-safe. ExpiryService diagnostics must use atomic reads for tick timestamp (Interlocked or lock).
- **No breaking changes to `/api/health` response shape.** Existing consumers (load balancers checking `status`) must keep working; the new `checks` object is purely additive.
</constraints>

<risks>
## Open risks for researcher/planner to validate

- **R-01:** SMTP TCP-connect probe can block IIS worker threads if `SmtpSettings.Host` is reachable but unresponsive. The 3 s timeout is a hard deadline — researcher should confirm `TcpClient.ConnectAsync(CancellationToken)` honors this on net10.0-windows.
- **R-02:** `dotnet list package --vulnerable --include-transitive` output format differs between SDKs. Researcher must pick a machine-parseable format (likely `--format json` if available in net10 SDK) and pin the exact parsing.
- **R-03:** `npm audit --json` output schema changed across npm versions. Lock the parser to the npm version CI uses; document the pin in `ci.yml`.
- **R-04:** `AdPasswordPolicyPanel` current render path may depend on an auth-aware bind. Verify the FEAT-002 policy endpoint works for anonymous/pre-change requests; if it doesn't, D-17's "no FGPP lookup" may already be the current state — then STAB-021 becomes purely CSS/layout.
- **R-05:** Installer Pester test coverage may not exist. If not, document STAB-019 verification as manual UAT and do NOT add a Pester toolchain in phase 10 — deferred.
</risks>
