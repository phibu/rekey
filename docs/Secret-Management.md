# Secret Management

PassReset handles several sensitive configuration values that must be protected in production deployments.

## Secrets Inventory

| Secret | Configuration Path | Environment Variable | Used By |
|--------|--------------------|---------------------|---------|
| LDAP bind password | `PasswordChangeOptions:LdapPassword` | `PasswordChangeOptions__LdapPassword` | AD password change provider |
| SMTP relay password | `SmtpSettings:Password` | `SmtpSettings__Password` | Email notification service |
| reCAPTCHA private key | `ClientSettings:Recaptcha:PrivateKey` | `ClientSettings__Recaptcha__PrivateKey` | Server-side CAPTCHA verification |

## Default Behavior

By default, secrets are stored in `appsettings.Production.json` on the IIS server. This file is:
- Generated by `Install-PassReset.ps1` on first install with empty placeholder values
- Readable only by the IIS app pool identity and local administrators (NTFS ACLs set by the installer)

To verify file permissions:
```powershell
Get-Acl C:\inetpub\PassReset\appsettings.Production.json | Format-List
```

## Environment Variables and User Secrets (STAB-017)

PassReset relies on ASP.NET Core's default configuration precedence. The three secrets above can be sourced from environment variables (production) or `dotnet user-secrets` (development) without any code changes — the `__` (double underscore) path delimiter is the built-in binding convention (D-16). No custom `PASSRESET_` prefix is introduced.

**Source precedence** (later wins):
1. `appsettings.json`
2. `appsettings.{Environment}.json`
3. `secrets.dat` (Phase 13 admin UI, encrypted via ASP.NET Core Data Protection)
4. `dotnet user-secrets` (Development only, when `UserSecretsId` is set in the csproj)
5. `AddEnvironmentVariables()` with `__` path delimiter
6. Command-line args

**Config key → env var mapping:**

| Config key | Env var |
|---|---|
| `SmtpSettings.Password` | `SmtpSettings__Password` |
| `PasswordChangeOptions.ServiceAccountPassword` | `PasswordChangeOptions__ServiceAccountPassword` |
| `ClientSettings.Recaptcha.PrivateKey` | `ClientSettings__Recaptcha__PrivateKey` |

**Developer workflow (user-secrets, Development environment only):**
```bash
cd src/PassReset.Web
dotnet user-secrets init
dotnet user-secrets set "SmtpSettings:Password" "dev-pass"
dotnet user-secrets set "ClientSettings:Recaptcha:PrivateKey" "test-key"
dotnet user-secrets list
dotnet user-secrets remove "SmtpSettings:Password"
```

**Operator workflow (AppPool environment variable via `appcmd`):**
```powershell
& "$env:windir\system32\inetsrv\appcmd.exe" set config `
    -section:applicationPools `
    "/[name='PassReset'].environmentVariables.[name='SmtpSettings__Password',value='<secret>']" `
    /commit:apphost
Restart-WebAppPool -Name 'PassReset'
```

**Regression guard:** `ClientSettings.Recaptcha.PrivateKey` is marked `[JsonIgnore]` on the DTO and never appears in `GET /api/password` regardless of whether the value was sourced from `appsettings`, user-secrets, or an env var (proved by `EnvironmentVariableOverrideTests`).

**Operator note:** The installer does NOT set these environment variables (D-18 — operator owns secret injection). Full encrypted-at-rest solutions (DPAPI/Key Vault) are scheduled for v2.0 (V2-003); STAB-017 is the documented stepping stone that unblocks externalizing secrets today without committing to a specific KMS.

## Hardening Options

### Option 1: Environment Variables (Recommended)

ASP.NET Core reads environment variables using the `__` (double underscore) separator. Environment variables take precedence over `appsettings.json` values, avoiding storing secrets on disk entirely.

**Setting environment variables via IIS Manager (GUI):**
1. Open IIS Manager > Application Pools > PassResetPool > Advanced Settings
2. Under Process Model, click the `...` button next to Environment Variables
3. Add each variable name and value from the table above

**Setting environment variables via PowerShell:**
```powershell
Import-Module WebAdministration

$poolName = "PassResetPool"
$envVars = @{
    "PasswordChangeOptions__LdapPassword"       = "YourLdapPassword"
    "SmtpSettings__Password"                    = "YourSmtpPassword"
    "ClientSettings__Recaptcha__PrivateKey"      = "YourRecaptchaKey"
}

foreach ($kv in $envVars.GetEnumerator()) {
    # Remove existing entry if present
    $path = "IIS:\AppPools\$poolName"
    $existing = Get-ItemProperty $path -Name environmentVariables.Collection `
        -ErrorAction SilentlyContinue |
        Where-Object { $_.name -eq $kv.Key }

    if (-not $existing) {
        Add-WebConfigurationProperty `
            -PSPath "MACHINE/WEBROOT/APPHOST" `
            -Filter "system.applicationHost/applicationPools/add[@name='$poolName']/environmentVariables" `
            -Name "." `
            -Value @{ name = $kv.Key; value = $kv.Value }
    }
}

# Recycle the app pool to pick up the new variables
Restart-WebAppPool -Name $poolName
```

**Verifying environment variables are set:**
```powershell
(Get-ItemProperty "IIS:\AppPools\PassResetPool").environmentVariables.Collection |
    Select-Object name | Format-Table
```

### Option 2: Skip LDAP Credentials Entirely (Simplest)

When `UseAutomaticContext: true` (the default for domain-joined servers), no LDAP username or password is needed. The application uses the computer account's domain context, which is the most secure option as no credentials are stored anywhere.

This eliminates the `LdapPassword` secret. SMTP and reCAPTCHA secrets still need to be managed if those features are enabled.

### Option 3: Windows DPAPI (Advanced)

For servers that cannot use environment variables, encrypt the production config file using Windows DPAPI. This ties the decryption key to the machine or user account:

```powershell
# Encrypt the config file so only the current machine can decrypt it
$content = Get-Content C:\inetpub\PassReset\appsettings.Production.json -Raw
$bytes = [System.Text.Encoding]::UTF8.GetBytes($content)
$encrypted = [System.Security.Cryptography.ProtectedData]::Protect(
    $bytes, $null, [System.Security.Cryptography.DataProtectionScope]::LocalMachine)
[System.IO.File]::WriteAllBytes("C:\inetpub\PassReset\appsettings.Production.json.dpapi", $encrypted)
```

Note: This approach requires a custom configuration provider in the application to decrypt at startup. It is not built-in and is only recommended for organizations with specific compliance requirements that prohibit both plaintext config and environment variables.

### Option 4: Admin UI with encrypted storage (Phase 13)

The Phase 13 admin UI stores secrets in `secrets.dat`, encrypted via ASP.NET Core Data Protection. On Windows, the DP key ring is itself protected by DPAPI — a Phase 13 realization of what Option 3 sketched.

To use it:
1. RDP to the server.
2. Browse `http://localhost:5010/admin`.
3. Enter secrets on the relevant pages.
4. Recycle the app pool.

Environment variables (Option 1) continue to override anything the admin UI writes — this is the intended STAB-017 precedence.

See `docs/Admin-UI.md` for the full operator guide.

## Credential Rotation

When a secret changes (e.g., LDAP service account password is rotated in AD):

1. **If using environment variables**: Update the IIS app pool environment variable via the PowerShell commands above, then recycle the pool.
2. **If using appsettings.Production.json**: Edit the file directly, then recycle the app pool:
   ```powershell
   # Edit the config file
   notepad C:\inetpub\PassReset\appsettings.Production.json

   # Recycle to pick up changes (no downtime — IIS overlaps old/new worker)
   Restart-WebAppPool -Name PassResetPool
   ```
3. **Verify**: After recycling, test a password change to confirm the new credentials work.

There is no downtime during credential rotation — IIS overlaps the old and new worker processes during a recycle.

## What Is NOT a Secret

The following values are safe to include in configuration files:
- `ClientSettings:Recaptcha:SiteKey` — this is the public site key, visible in the browser
- `PasswordChangeOptions:DefaultDomain` — the AD domain name
- `PasswordChangeOptions:LdapHostnames` — the LDAP server hostnames
- `SmtpSettings:Host` — the SMTP relay hostname

## Security Controls

- `Recaptcha.PrivateKey` is marked `[JsonIgnore]` on the `ClientSettings` model, preventing it from being returned by `GET /api/password`
- `appsettings.Production.json` and `secrets.json` are in `.gitignore`
- The installer sets NTFS permissions restricting file access to the app pool identity
- Environment variables are scoped to the specific IIS app pool, not system-wide
